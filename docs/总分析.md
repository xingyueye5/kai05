## 1. 当前项目核心方法回顾（一句话）

> Kai05-VLA = π0/π0.5 (PaliGemma + Action Expert + Flow Matching) +
> 「**VC-Value (kNN on SigLIP features) → Advantage 离散化 → 作为 task prompt 注入** 」+
> 可选 TD-Learning Value Head + 推理时 CFG。

关键代码模块（后续会反复引用）：

| 模块 | 角色 |
|---|---|
| `scripts/extract_siglip_features.py` | SigLIP2 帧特征提取 |
| `scripts/calculate_VC_value.py` | kNN-based value 估计（GPU 多进程） |
| `scripts/calculate_lerobot_advantage.py` | reward + advantage 离散化 + 写入 `task_index` / `tasks.jsonl` |
| `src/openpi/models_pytorch/pi0_pytorch.py` | advantage sincos embedding + value head + TD + CFG |
| `src/openpi/training/custom_lerobot_dataset.py` | dataset 包装，输出 `action_advantage` |
| `scripts/train_pytorch.py` | DDP 训练入口 |
| `scripts/serve_policy.py` | WebSocket 推理服务 |

## 2. 顶会相关论文调研与创新点分析

围绕「advantage / value 信号来源 + flow-matching VLA 后训练」这条主线，我们梳理与本工作最相关的 11 个顶会工作，分四类。

### 2.1 同范式（advantage-conditioned VLA）—— 必须超越或区分的"近邻竞品"

#### [1] π*0.6 (RECAP), Physical Intelligence, arXiv 2511.14759, 2025.11
- **核心 idea**：在 π0.5 流匹配 VLA 上做 advantage-conditioned offline RL：训练 670M value model 估计 V(s)，advantage = 即时 reward + γV(s') − V(s)，离散化后作为 prompt 条件注入；推理时设 advantage=high。RECAP 三阶段：offline RL 预训练 → SFT → on-policy 干预微调。
- **强**：奠定"advantage conditioning 是 flow-matching VLA 后训练正确范式"；real-world 大规模验证。
- **依赖**：
  1. 人工 reward 标注；
  2. 训练独立 value model（670M）；
  3. 实物机器人 rollout + 人在环干预。
- **与本项目的关系**：Kai05-VLA 的 VC-Value 等于 π*0.6 中 value model 的**无训练替代品**；advantage 注入机制几乎一致。**这是我们最重要的对照基线。**

#### [2] FlowPRO (RPRO), arXiv 2606.05468, 2026.06
- **核心 idea**：reward-free 流匹配 VLA 离线 RL；用专家 demo 与人在环干预数据构造 preference pair，对 flow policy 做 proximalized preference optimization（RPRO loss）。
- **强**：第一篇明确证明 "VLA 后训练可以无 reward"。
- **依赖**：依然需要"干预数据"或 preference pair，仍非纯静态公开数据。
- **与本项目的关系**：FlowPRO 把 reward 去掉了；我们在 FlowPRO 之后**再去掉 preference 和干预数据**，这是一个清晰可写的 niche gap。

#### [3] FPO（Flow-matching Policy Optimization, 中科院自动化所, 2025.10）
- **核心 idea**：用无似然代理替代 PPO 中显式 log-prob，在 flow-matching VLA 上完成 online RL 微调。
- **与本项目的关系**：online RL 路线，与"仅公开静态数据"目标正交，**仅作 related work 引用**，凸显我们是 offline / data-only 路线。

### 2.2 关键技术先验（advantage 信号源头）

#### [4] Decision Transformer (NeurIPS 2021)
- 用 return-to-go 作为 sequence model 的 condition，把 RL 转成 supervised sequence modeling。**advantage-conditioning 的思想源头**。
- 与我们的差别：DT 需要 reward；我们 reward-free，且基于流匹配而非自回归 Transformer。

#### [5] AWR / RvS / IQL (ICLR 2022–2023)
- 离线 RL 把 advantage 用作 BC 权重或条件。
- 与我们的差别：他们需要训练独立 value 网络，且大多在 state-based 任务。

#### [6] HER — Hindsight Experience Replay (NeurIPS 2017)
- 用 episode 后段实际到达状态做 goal relabeling。
- **与我们的关系**：本项目 plan v3 的 RTAP（Reverse-Time Anchored Progress）本质是 HER 在 foundation-feature 空间的 **soft 版本**——把"goal=episode 终点帧的特征"，progress=向终点的视觉距离衰减。这是写论文时定位 RTAP 的核心 anchor 工作。

#### [7] Goal Sets, Not Goal States (arXiv 2606.09476, 2026.06)
- 单一 goal state 过约束；用 goal *set* 做 relabeling。
- **启发**：plan v3 的 Multi-Anchor RTAP（kmeans 多 goal）正是这条线，需要在 paper 里 cite。

### 2.3 关键技术先验（flow matching 自身的不确定性）

#### [8] AdaFlow（NeurIPS 2024）★★★ 与我们最容易被混淆的工作
- **核心 idea**：观察到 flow-matching 输出的 action chunk 在不同 noise 下的 variance 是有意义的信号——**用它在推理时自适应调度去噪步数**（高 variance 多步，低 variance 少步）。
- **与我们的精确区分（必须在论文 Section 2 单独列）**：

  | | AdaFlow | 本项目（FlowVar） |
  |---|---|---|
  | variance 用途 | **推理时**调度步数（"做快"） | **训练时**作 advantage 条件信号（"做好"） |
  | 是否做 advantage conditioning | ❌ | ✅ |
  | 论文目标 | 加速 | 提升策略质量 |

  → 一句话：**AdaFlow 用 variance 做"快"，FlowVar 用 variance 做"好"**。这是我们 contribution 2 抗审稿的关键陈述。

#### [9] VFP — Variational Flow-Matching Policy (arXiv 2508.01622)
- VAE + flow matching 处理 multi-modal demo。提示我们：flow-matching variance 与 multi-modality 的关联是已被讨论的话题，但**没人把它做成 advantage 信号**。

### 2.4 表征/数据侧支撑

#### [10] Diffusion Policy + CFG（CFG-DP, ICTAI 2025；arXiv 2510.09786）
- 已经把 CFG 引入到 diffusion policy；为我们论文里"在 flow matching VLA 上做 CFG"提供了直接的 related work 锚点（但 CFG-DP 没有 advantage 信号源的创新）。

#### [11] DmC — Nearest-Neighbor Guidance Diffusion (arXiv 2507.20499, 2025)
- 用 k-NN 在跨域 offline RL 中做 reward shaping。**强化"kNN-in-feature-space 做 value-related 估计"这条线在 ICML/ICLR 已是被认可路径**，VC-Value 在范式上不孤立。

## 3. 当前方法被审稿人击中的 4 个核心弱点（来自 explain.md / paper.md）

1. **W1 — VC-Value 的根本性循环依赖**：靠 cosine 相似度从相似帧的 progress 取均值，但要"限制 progress window 内匹配"——已经假设了 progress 已知，违反 value function 的因果性。
2. **W2 — 强依赖 progress_gt 标注**：当前 advantage_source 默认是 `progress_predicted` 或 `progress_gt`，但 progress_gt 来自人工/启发式标注，与"公开数据集"的卖点冲突。
3. **W3 — 泛化性差**：cosine-similarity-in-SigLIP-feature-space 假设视觉相似 ⟹ value 相似，跨任务跨场景一致性弱。
4. **W4 — 创新点偏组合 trick**：SigLIP + kNN + advantage cond. + CFG，每一项都有先例，缺乏算法层面的新单元。

## 4. 最终方法定位（直接采用 plan.md v3 路线，并做精化）

### 4.1 一句话定位

> **既不要 reward、preference，也不要 rollout、干预——
> 仅用公开静态 demo，从（i）视觉特征对 episode 终点的距离 + （ii）flow-matching policy 自身的 action variance
> 两个相互正交的内部信号中提取 advantage，完成 π0.5 风格 flow-matching VLA 的离线后训练。**

### 4.2 三个核心 contribution

| 编号 | 名称 | 解决的 W | 与最近邻竞品的差异 |
|---|---|---|---|
| **C1** | **RTAP**（Reverse-Time Anchored Progress）：用 episode 最后一帧作为 goal anchor，在 SigLIP 特征空间衡量当前帧到 goal 的距离，直接得到 progress 估计；多任务时做 Multi-Anchor (kmeans on goals) | W1 + W2 | vs π*0.6：不训练 value model；vs HER：在 feature space 做 soft relabel；vs 原 VC-Value：无循环依赖、无 progress_gt 依赖 |
| **C2** | **FlowVar Advantage**：用 vanilla fine-tuned π0.5 在每个 state 上做 N=8 次不同 noise 的 flow 采样，取 action chunk 标准差作为 advantage 信号（方差越小 advantage 越高） | W3 + W4 | vs AdaFlow：用于训练 advantage 而非推理调度；vs π*0.6：不需要外部 value model；**全新的内部信号** |
| **C3** | **Orthogonal Signal Fusion**：RTAP 是 input-side（视觉），FlowVar 是 output-side（动作），二者经实证正交；融合后在 RTAP 失效的 long-horizon 任务 / FlowVar 失效的 cold-start 任务上互相救场 | W3 + W4 | 全新的信号组合；理论可证明正交（Proposition 3） |

### 4.3 把它放回竞品 landscape

| 方法 | reward | value model | preference | rollout | 干预 | 数据来源 |
|---|:-:|:-:|:-:|:-:|:-:|---|
| π*0.6 | ✅ | ✅ | ❌ | ✅ | ✅ | 自采 + 干预 |
| FlowPRO | ❌ | ❌ | ✅ | ✅ | ✅ | 真实 robot |
| AdaFlow | — | — | — | — | — | 公开 demo |
| **本工作 (FreeVAC)** | **❌** | **❌** | **❌** | **❌** | **❌** | **公开静态数据集** |

Niche 留得很干净，足以支撑一篇顶会论文的 positioning。

## 6. 改动思路（按 4 个阶段）

### 阶段 1：数据层 — 给所有公开数据集打上三种 advantage 标签

**目标产物**：每个 LeRobot 数据集的 parquet 文件多出 3 列：`rtap_progress`、`flowvar_advantage`、`fused_advantage`，以及由这些列经现有 `calculate_lerobot_advantage.py` 离散化后的 `task_index`。

**关键改动思路**：

1. **RTAP（C1）**：先用现有 `extract_siglip_features.py` 提取所有帧 SigLIP2 特征（与原 VC-Value pipeline 共用，零额外成本）→ 新脚本对每个 episode 取最后帧 feature 做 anchor，计算余弦距离归一化为 progress。Multi-Anchor 版本用 sklearn kmeans 聚类 episode 终点特征。
2. **FlowVar（C2）**：先用 `train_pytorch.py` 在目标数据集上做一次 vanilla π0.5 fine-tune（不带 advantage 条件）→ 得到 reference policy `π_ref` → 新脚本批量调用 `π_ref.sample_actions(state, noise=ε_i)` 8 次，计算 action chunk 在 sample 维度上的 std，聚合为 advantage。这一步是**关键的额外开销**（~50–80 GPU·h，一次性）。
3. **Fusion（C3）**：percentile-normalize 三种信号到 [0,1]，加权平均（默认 α=0.5），写回 parquet。
4. **离散化**：直接调用现有 `calculate_lerobot_advantage.py --advantage_source fused_advantage --advantage_type 5bins`（仅需新增一个合法值，逻辑不变）。

### 阶段 2：训练层 — 复用现有 pipeline，仅切换 advantage 来源

**关键认知**：**模型代码完全不改**。advantage 信号变化对模型来说只是 `tasks.jsonl` 里的 prompt 字符串（"... advantage: 0.6"）变了和 `task_index` 列对应不同的 advantage bin。`pi0_pytorch.py` 中 `embed_prefix` 拿到的 `action_advantage` 数值依然是 `(bin+1)/N`，sincos embedding 自然适配。

每个实验只是一份新的 `configs/train/*.yaml`：
- `freevac_rtap_only_libero_spatial.yaml`
- `freevac_flowvar_only_libero_spatial.yaml`
- `freevac_fusion_libero_spatial.yaml`
- baseline: `pi0_vanilla_libero_spatial.yaml`、`pi06_trained_value_libero_spatial.yaml`（用现有 value_head + TD_learning 路径作 π*0.6 风格 baseline）

### 阶段 3：推理与评估 — 复用现有 serve_policy + CFG

LIBERO 评估用新增 `scripts/benchmark_libero/run_libero_eval.py`：起一个 in-process policy 模拟 `serve_policy.py` 的接口（避免 WebSocket 跨进程开销），在 LIBERO simulator 中跑 N 个 episode，统计 success rate。CFG inference 完全复用 `sample_actions` 中的 cfg 分支。

### 阶段 4：分析与写作

- **正交性 figure**（C3 的命根子）：`scripts/analysis_orthogonality.py` 输出 RTAP-adv vs FlowVar-adv 散点图 + Pearson correlation；如果 |ρ| < 0.6 → contribution 3 成立；否则启动 Plan B。
- **失效分析 figure**：在 LIBERO-Long 上找出 RTAP 给高 advantage 但 rollout 失败的状态、FlowVar 给高 advantage 但 rollout 失败的状态，可视化对比 → 证明二者失效模式互补。

## 8. Gate Criterion 与 Plan B（Week 3 末决策）

满足 ≥ 3/5 继续主路线，否则切 Plan B：

| 检查 | 阈值 | 含义 |
|---|---|---|
| π0.5 vanilla 复现 LIBERO-Spatial | ±2% 官方数 | setup 正确 |
| RTAP 与 oracle progress Spearman | > 0.5 | RTAP 有效 |
| FlowVar 与 oracle progress Spearman | > 0.3 | FlowVar 有意义 |
